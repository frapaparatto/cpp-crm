**What I learned:**

- Pass-by-value-then-move is a pessimization for setters. When a setter assigns to an existing member, the pass-by-value pattern always allocates a new buffer for the parameter, moves it into the member, and discards the member's old buffer. With `const&`, the assignment operator can reuse the existing buffer if capacity is sufficient. The benefit is most visible for strings exceeding the SSO threshold since short strings store data inline and never heap-allocate. Constructors keep pass-by-value-then-move because the member does not exist yet in the initializer list, so there is no buffer to reuse.
- Rule of Five: if a class manages a raw resource, the compiler cannot generate correct copy/move/destruction behavior. You must define all five special functions. Declaring any one prevents the compiler from generating the others. Use `= default` to re-enable compiler generation for specific functions.
- Empty destructors break trivial copyability. The compiler sees a user-defined destructor and cannot prove there are no side effects, so `std::copy` falls back to element-by-element copying instead of `memmove`. Never write an empty destructor. Use `= default` or omit it entirely.
- Error handling strategy: asserts for programming errors (fires when the programmer made a mistake, removed in release builds via `NDEBUG`), exceptions for rare runtime errors (I/O failures, constraint violations, zero cost on success path, expensive only when thrown), `std::optional` for expected absence (a query returning no result is not an error). The practical test: if the check fails, does it mean the programmer made a mistake? If yes, assert. If the user caused it, handle with exceptions or return types.
- Design by Contract: preconditions are the caller's responsibility, postconditions are the function's responsibility, invariants must always hold true. A moved-from object should be left in the same state as a default-constructed object to keep invariants consistent. If defining a class invariant is hard, the class likely has too many responsibilities.
- Debug vs release builds: debug mode (`-O0 -g`) enables asserts and debug symbols, no optimization. Release mode (`-O2 -DNDEBUG`) enables optimization and removes all asserts. Asserts can be used freely during development to enforce invariants with zero cost in production.
