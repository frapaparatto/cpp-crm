**What I learned:**
- constexpr std::array vs static const std::vector for compile-time known data: array stores inline with no heap allocation, vector heap-allocates on first call. Use array when size and values are known at compile time.
- static constexpr vs constexpr for local variables: static guarantees one instance across all calls, constexpr alone may create a new stack instance per call (though compilers typically optimize this away). static makes the intent explicit.
- std::string_view as parameter for isValidDate was rejected because stoi requires std::string internally, so starting with string_view would force conversions. const std::string& avoids unnecessary conversion when the input already comes from std::getline.
- Structured bindings (auto& [name, controller]) destructure std::pair from map entries into named variables. C++17 feature.
- Save as an application-level concern that iterates over all controllers calling save() on each, rather than requiring the user to save per entity. Each controller forwards save() to its own repository.

**Architecture reasoning:**
- Status field is always set on the entity, never empty. The optionality exists only in the DTO (ClientData, PolicyData) to signal "user didn't choose, apply default." The service layer uses value_or() to apply the default. This separates user intent (skip or choose) from domain invariant (status always has a value).
- Decided against a static counter for policy numbers because it resets on program restart and doesn't know the last number after loading from CSV. Using UUID instead, consistent with Client. Policy deduplication deferred to polish phase.
- Date validation regex checks format, then substr+stoi extracts year/month/day for calendar validation. Regex and manual checks overlap (both verify structure) but regex catches malformed input early before attempting numeric parsing.
