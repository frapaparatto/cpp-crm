**What I built:**
- Applied `std::string_view` to read-only string parameters across client-side code: `contains()` in strops, `statusFromString`, `removeClient`, `findByUuid`, `findByEmail`, validation helpers `isValidEmail`, `isDigitsOnly`, `isValidPhone`

- Left `deserialize()` as `const std::string&`: `std::getline` with `std::stringstream` requires `std::string`, changing the parameter would force an internal conversion adding cost rather than removing it

**What I learned:**

`std::string_view` is a non-owning read-only view of a contiguous sequence of characters. It stores only a pointer and a size. No heap allocation, no ownership, no copy of the data. It accepts `std::string`, string literals, and char arrays without constructing a temporary object.

The decision rule for each `const std::string&` occurrence:
- function only reads the string and does not store it: change to `string_view`
- function stores the string into a member or passes it to a constructor: keep `const std::string&` or take by value
- getter returning a reference to a member: never change to `string_view`, the member owns the data and the reference is valid for the object's lifetime

`std::regex_match` does not accept `string_view` because it predates C++17. Use the iterator overload `regex_match(view.begin(), view.end(), pattern)` to avoid constructing a temporary `std::string` inside the function.

`std::string` passed to a `string_view` parameter converts implicitly and safely. `string_view` has an implicit constructor from `std::string`. No changes needed at call sites when updating a parameter from `const std::string&` to `string_view`.

**Key reasoning on `deserialize()`:**

`deserialize` takes `const std::string& line` and uses `std::stringstream ss(line)` internally. `std::stringstream` requires `std::string` for construction. Changing the parameter to `string_view` would require converting back to `std::string` inside the function, adding a construction cost that did not exist before. The change would make performance worse, not better. Left unchanged.

**Note on motivation:**

Some of these changes provide minimal measurable improvement at the current data scale since `const std::string&` already avoids copies. They were applied for learning purposes and to establish the correct pattern before the codebase grows.
