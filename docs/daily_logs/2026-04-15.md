**What I learned:**
- `std::optional<std::string>{""}` and `std::nullopt` are distinct states. An empty string implicitly constructs an optional with `has_value() == true`. Only `std::nullopt` represents absence. The round-trip bug was caused by this implicit construction, serializing absence as `""` and deserializing it back as a present optional holding an empty string.
- `std::optional<std::string>{s}` is construction, not conversion. The `{}` uniform initialization syntax constructs the optional directly from the string value. Equivalent to `std::optional<std::string>(s)` but preferred because `{}` prevents narrowing conversions.
- The fix was kept local to `deserialize()` rather than changing the serialization format. This was a deliberate choice, existing CSV files remain valid, and the invariant "empty string in CSV means nullopt" is sufficient for this project.
