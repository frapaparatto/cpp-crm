**What I learned:**
- The search vs view distinction clarifies user intent: search shows a results list (zero, one, or many), view resolves to exactly one entity and shows full detail. Conflating the two forces resolution where the user might just want to browse results.
- Display-only data duplication (pricing table in controller mirroring the service table) is acceptable when the authoritative source stays in the service. The controller preview is read-only convenience; the service does the real calculation.
- resolveClient as a free function taking ClientService& is dependency injection applied to a free function: the function needs search capability, receives it as a parameter, doesn't own or store it.

**Architecture reasoning:**

- Chose to extract cli_helper free functions over three rejected alternatives: duplicating resolution logic in PolicyController (obvious duplication), injecting ClientController into PolicyController (breaks controller independence), or adding resolution methods to the abstract IEntityController (forces all controllers to implement client-specific logic). Free functions with service parameters keep each controller independent while sharing the resolution pipeline.
- Split cmdSearch and cmdView into distinct operations. Previously cmdSearch resolved to one client and displayed detail, conflating two user intents. Now cmdSearch displays a results list with optional drill-down (acknowledged as slightly blurring the line, TODO left for future strictness). cmdView always resolves to exactly one and shows detail.
- PolicyController receives ClientService& in its constructor specifically to pass it to the resolveClient free function. This is not a layer violation: the controller orchestrates user interaction, which legitimately includes selecting a client before creating a policy.
