**Active branch:** feature/3-policy-crud
**Issue:** Policy CRUD

**What I built:**
- Created `policy_status.hpp/cpp` in `src/domain/` with four converters: `policyStatusToString`, `policyStatusFromString`, `policyTypeToString`, `policyTypeFromString`, mirrors the `client_status.hpp/cpp` pattern
- Added `getCreatedAt()` and `getUpdatedAt()` getters to `Policy`, they were missing, blocking the serialize side from outputting those fields
- Implemented `CsvPolicyRepository::serialize` and `CsvPolicyRepository::deserialize`: full 10-field CSV round-trip (uuid, client_uuid, policy_type, start_date, end_date, amount, policy_status, notes, created_at, updated_at), using `std::to_string` and `std::stod` for the double amount field
- Refactored `IPolicyRepository` interface: signatures updated to `string_view`, pass-by-value Policy with move semantics, `const` on all read methods, `save()` added, `findAll()` returns `const std::vector<Policy>&`

**What I learned:**
- `std::to_string` and `std::stod` are the standard way to round-trip a double through a string. `stod` throws `std::invalid_argument` on bad input, which fits the same error contract used in `statusFromString`
- Extracting a lambda to a shared utility only makes sense once a second caller appears: `toOpt` was fine as a local lambda in client, but policy needing the same logic was the right trigger to move it out

**Architecture reasoning:**
- `stringToOptional` was initially placed in `csv_utils.hpp` under `insura::data` as a header-only one-liner with `inline`. On reflection, a dedicated file for a single utility was not worth it; the function is general enough to live in `utils.hpp/cpp` alongside the other domain utilities, where it is available to any layer without creating a data-layer dependency
- `policy_status.hpp/cpp` follows the same structure as `client_status.hpp/cpp` intentionally. The enum-to-string strings ("active", "expired", "auto", "life") are canonical CSV identifiers that live in the domain layer. If they lived in the data layer they would be invisible to any future display or validation logic that doesn't go through the repository
