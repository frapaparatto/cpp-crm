**May 4, 2026**

**Active branch:** feature/9-policy-crud
**Issue:** Policy CRUD

**What I built:**
- Completed `PolicyService` with five methods: `addPolicy`, `deletePolicy`, `editPolicy`, `searchByClient`, `searchPolicy`
- `addPolicy` validates start date via `utils::date::isValidDate`, derives `end_date` via `calculateEndDate`, computes `amount` from a static pricing table, defaults status to `PENDING` if not supplied by caller
- `calculateAmount` uses a `constexpr double[4][4]` pricing table (4 types × 4 durations: 6/12/24/36 months). A `static_assert` guards row count against enum size drift
- `editPolicy` is intentionally narrow: only `policy_status` and `notes` can be updated; all other fields are immutable contract-defining fields per ADR-019
- Build passes clean in `build-debug`

**Architecture reasoning:**
- `addPolicy` does not accept `end_date` or `amount` from the caller. Both are derived server-side: `end_date` from start date + duration, `amount` from the pricing table. This keeps the service as the single source of truth for derived contract data and prevents callers from injecting inconsistent values
- `editPolicy` accepting only `PolicyData` (same DTO used by `addPolicy`) is a deliberate reuse. The service reads only the fields it cares about (`policy_status`, `notes`) and ignores the rest. This avoids a separate `EditPolicyData` struct while keeping the edit contract explicit in code comments
- `deletePolicy` calls `findByUuid` before `removePolicy` so it throws a meaningful error if the UUID is unknown. The repository's `removePolicy` is silent on missing UUIDs (just a no-op), so the guard must live at the service layer

**What I learned:**
- A `constexpr` 2D array indexed by enum cast to `int` is a clean, zero-overhead dispatch table for small finite mappings. The `static_assert` on row count is the right guard: it will catch a mismatch at compile time if a new `PolicyType` is added without updating the table
