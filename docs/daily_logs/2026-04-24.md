**What I learned:**
- `constexpr` guarantees compile-time evaluation. `const` only guarantees immutability but the initialization can happen at runtime. `constexpr` implies `const` but not the other way around.
- A `constexpr` function is dual-purpose: it evaluates at compile time when the context demands it (e.g., initializing a `constexpr` variable) and at runtime otherwise. The call site decides, not the function. `consteval` is the stricter variant that forces compile-time-only evaluation.
- IIFE (Immediately Invoked Function Expression) is a lambda defined and called in the same statement with `()`. The return value initializes a variable directly, allowing it to be `const` even when the value depends on branching logic.
- Three solutions for complex initialization ranked by scope: ternary for simple two-way choices, IIFE for local multi-branch logic that doesn't justify a name, named function for reusable or complex logic.
- In C++ declaration is construction. Writing `std::string s;` calls the default constructor, then `s = "value";` calls the assignment operator. That is two operations. Writing `const std::string s = "value";` calls the constructor once with the value directly. Variables should not be declared until they can be initialized with a meaningful value, and should be `const` whenever they are not modified after initialization.
- Member initializer lists are evaluated in the order fields are declared in the class, not in the order they appear in the initializer list. If the written order disagrees with declaration order and one member depends on another, the dependent member reads from an uninitialized or moved-from source. The code compiles and runs but silently produces wrong results.
- In-class initializers act as fallbacks. A constructor's initializer list overrides them when needed. `const` members and reference members can only be initialized in the member initializer list or via in-class initializer.
- `noexcept` is a runtime contract that no exception will propagate out of the function. Not verified at compile time. If violated, `std::terminate` is called. `std::vector` checks whether the move constructor is `noexcept` to decide between moving (fast) and copying (safe fallback) during reallocation.

**Architecture reasoning:**

- Evaluated the IIFE pattern against three locations in the codebase. Applied it to main.cpp because repo was declared uninitialized and assigned inside if/else branches, preventing it from being `const`. The lambda wraps the startup menu logic and returns the constructed repo directly, so the variable is `const` and initialized in a single expression.

- Rejected it for `CsvClientRepository::load()` because `clients_` is a member variable and cannot be `const`. The main benefit of the pattern does not apply. The current approach (build a temporary vector, validate, move into `clients_`) is already clean and the separation between building the data and committing it is explicit. Wrapping the same logic in a lambda would add indirection for no gain.

- Rejected it for `promptEditData` prompt strings because the existing ternary is already readable for a simple two-way choice. The IIFE would be justified if the condition grew beyond two branches, but currently it would add complexity without improving clarity.
