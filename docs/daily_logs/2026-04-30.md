**What I learned:**
- Object slicing is why you need unique_ptr for polymorphic ownership. A derived object stored by value as a base type loses its derived data. unique_ptr preserves the full object on the heap and the base interface provides access without slicing.
- `std::make_unique<Derived>(args)` allocates on the heap and constructs in one call. The returned unique_ptr<Derived> converts implicitly to unique_ptr<Base> when stored in a container typed to the base interface.
- `.get()` on a unique_ptr returns the raw non-owning pointer. Used for active_controller_ which observes without owning. The unique_ptr in the map still owns the object and handles destruction.
- `std::string_view` cannot be used with `std::unordered_map::find` in C++17 because heterogeneous lookup is not supported until C++20. Keep `const std::string&` for execute() since the input comes from std::getline which already produces a std::string.
- `std::function<void()>` means a callable that takes no parameters and returns nothing. Empty parentheses are the parameter list, void is the return type.
- Capturing `[this]` in a lambda captures the controller instance pointer so dispatch table entries like `commands_["add"] = [this]() { cmdAdd(); }` call the right object's method.
- CMake build types are controlled by CMAKE_BUILD_TYPE. Debug mode uses -g -O0 (no optimization, debug symbols, asserts active). Release mode uses -O2 -DNDEBUG (optimization enabled, asserts removed). NDEBUG is defined globally by the compiler flag, not manually in source files. The `<cassert>` header handles the ifdef internally. Both build directories can coexist side by side.
- Both the inner and outer while loops checking `running_` is idiomatic for nested interactive menus where exit should work from any level. The inner loop has an additional "back" condition the outer loop doesn't.
- A helper function returning bool to indicate "handled or not" is idiomatic dispatch for sharing command handling across multiple menu levels.

**Architecture reasoning:**

- Chose the orchestrator + controller pattern over prefix commands ("add-policy") and if/else branching on entity type. Prefix commands don't scale, if/else scatters entity logic across every handler. The controller interface is proportional: three entities with identical command sets justify a shared six-method interface. Each controller is self-contained with its own service, dispatch table, and helpers. Adding a new entity means one new controller class and one line in Application's constructor.
- Save, exit, and clear are application-level commands that stay in Application. CRUD commands belong in controllers. handleAppCmds() is called in both loops to avoid duplicating the application-level command handling.
- Identified and fixed a bug where typing "exit" in the entity menu would set running_ to false but the inner while(true) loop would not exit until the next iteration, causing a phantom prompt. Changed the inner loop from while(true) with a `!running_` break check to while(running_) so the loop condition catches the flag change immediately.
