**What I learned:**
- `std::transform` applies a unary operation to each element of an input range and writes results to an output range. When input and output are the same container, the transformation is in place. Returns an iterator past the last written element.
- `::tolower` and `::toupper` from `<cctype>` are the correct single-argument versions. `std::tolower` from `<locale>` takes two arguments and cannot be used as a simple character transform. The `::` prefix selects the global namespace explicitly.
- All `<cctype>` functions require `unsigned char` values. Plain `char` is signed on most platforms and values above 127 are negative, passing them is undefined behavior. The cast `static_cast<unsigned char>(c)` is not defensive programming, it is correct usage per the C standard.
- `static_cast<T>` is preferred over C-style cast `(T)` because it only performs well-defined conversions and refuses at compile time if the conversion is not valid. C-style casts are a blunt instrument that can perform reinterpret casts silently.
- The reverse iterator pattern for right-trim: `rbegin()` and `rend()` walk the string backward. `find_if` with `!isspace` finds the last non-space character. `.base()` converts the reverse iterator back to a forward iterator, offset by one, pointing to the position just after the found character, which is exactly where `erase` should start to remove trailing spaces.
- `std::remove_if` strips all matching characters from the entire string, not just the edges. Using it for trim would destroy spaces in the middle of the string. The two-pass `find_if` approach is necessary for leading and trailing whitespace only.
- `std::search` with a binary predicate slides a window of needle-size across the haystack, checking each position using the predicate. No string copies, no heap allocations. `std::string::find` has no hook for custom comparison, case-insensitive matching with `find` requires creating lowercase copies first, which means heap allocations per client per search call.
- `capitalize` uses a `next` boolean flag. `next` starts as `true`. After each space `next` becomes `true`. When `next` is true and the current character is not a space, it is uppercased and `next` is set to `false`. Multiple consecutive spaces are handled naturally, the flag stays true regardless of how many spaces are seen in a row.

**Architecture reasoning:**

Normalization lives in the CLI layer because the CLI is the point where raw human input enters the system. The service assumes it receives clean data. This contract is documented in `client_service.cpp`. A future non-CLI caller such as a batch importer would be responsible for its own normalization before calling the service. Notes and address fields are trimmed but not capitalized to preserve user intent for free-text fields.
