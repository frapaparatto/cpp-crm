**What I learned:**
- `std::cin >>` leaves a trailing newline in the buffer. `std::getline` does not skip leading whitespace and reads it immediately as an empty string. The fix is to avoid mixing `>>` and `getline` entirely, use `getline` everywhere and parse integers with `std::stoi`. This is consistent with the rest of the CLI and eliminates the need for `std::cin.ignore()`.
- `std::cin.ignore(std::numeric_limits<std::streamsize>::max(), '\n')` discards everything up to and including the first newline. `max()` means no practical character limit. The two-argument form is preferred over the default single-argument form which only discards one character.
- `static_cast<int>` and `static_cast<std::size_t>` are needed when comparing or indexing across signed and unsigned types. Mixing them produces compiler warnings and can cause subtle bugs when negative values wrap around. Explicit casts communicate intent and suppress the warning correctly.
- Validation exists correctly in two layers for different reasons: CLI layer for immediate UX re-prompt, domain layer as the last line of defense for any caller that bypasses the CLI. The duplication is intentional. Each copy serves a different master.
- `resolveClient()` correctly separates three concerns: prompting for a search term, calling the service, and disambiguating multiple results. Commands that need a single client call it as a shared pipeline. This avoids duplicating the search-select logic across cmdEdit, cmdDelete, and cmdSearch.
- `std::unique_ptr` in `main.cpp` solves the problem of needing to declare `repo` before the if/else block while not being able to default-construct it without a filepath. A pointer starts as nullptr and is assigned inside the branches. `make_unique<T>(args...)` is the safe construction form.
- `save()` declared private on `CSVClientRepository` but public on `IClientRepository` is still callable through the interface reference. C++ access control is checked against the static type at the call site, not the dynamic type.

**Architecture reasoning:**

Phase 1 review confirmed the layered architecture is correctly implemented. Domain enforces invariants. Service enforces business rules. CLI enforces UX. Data layer handles persistence. No upward dependencies exist. Composition root in `main.cpp` is the only place that knows about the concrete CSV type.

Two design decisions documented but not yet in ADRs: empty string as "not provided" sentinel for required fields in `ClientData` during edit (Option B from edit_flow reasoning), and `Application` calling `repo_.findAll()` directly bypassing the service for list. Both are defensible trade-offs for this project.

`cmdSearch` forcing resolution to a single client via `resolveClient()` is correct for edit and delete but potentially wrong for a pure search/display operation. ADR-014 captures this for a future decision.

**Critical bugs to fix before issue #4:**
- Notes serialization: `serialize()` always writes empty string, `getNotes()` is never called. Notes entered by the user are lost on save.
- Optional field round-trip: absent optionals serialized as `""` and deserialized back as `optional{""}`, `has_value()` returns true for all optional fields after one save-load cycle. All downstream `has_value()` checks are broken after the first restart.
